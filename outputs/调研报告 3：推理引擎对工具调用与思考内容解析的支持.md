# 调研报告 3：推理引擎对工具调用与思考内容解析的支持

本文围绕 Week 4 的调研主题，分析主流推理引擎在工具调用（function calling / tool use）与思考内容解析（reasoning content / thinking）上的支持方式。材料来源包括 OpenAI Function Calling、Anthropic Tool Use、XGrammar、vLLM / SGLang 源码与文档、Qwen / DeepSeek chat template 资料，以及本地 serving benchmark、工具调用专项 benchmark、grammar-guided generation 开销实验与启动行动笔记。

与前两份报告一致，本文不只比较功能清单，也结合 `benchmark_results/runs/20260616_025640_qwen35-sglang-vllm` 的 SGLang / vLLM serving 实测结果、`benchmark-vllm-v.s-sglang/results/summary/grammar_overhead_qwen25_7b_gpu*_20260702_2148.jsonl` 的 vLLM 工具调用专项结果，以及 `benchmark_results/2026-06-20-large-model-startup-action-notes.md` 中 14B、32B、GLM FP8 的启动结论，分析这些能力在真实部署时的约束。

## 摘要

工具调用并不是模型直接执行函数，而是模型生成结构化的调用意图：函数名、参数 JSON、调用 ID。真正的函数执行、权限控制、异常处理和结果回传仍由应用侧完成。OpenAI 和 Anthropic 的协议差异主要体现在消息格式上，但共同点都是把模型输出拆成“自然语言内容”和“工具调用块”。

推理引擎层的难点不是简单返回一个 `tool_calls` 字段，而是要解决三件事：第一，如何根据模型原生 chat template 把不同格式的工具调用解析成 OpenAI-compatible 结构；第二，如何在 streaming 模式下增量解析半截 JSON、XML 或特殊 token 包裹的工具调用；第三，如何把 `<think>...</think>`、analysis channel 或模型专用 thinking 标记拆到 `reasoning_content`，避免与最终答案或工具调用混在一起。

vLLM 的设计更接近“统一 OpenAI serving 层 + 可插拔 parser”。它通过 `--enable-auto-tool-choice`、`--tool-call-parser`、`--reasoning-parser` 等参数启用不同模型的 tool / reasoning parser，本地源码中已经包含 DeepSeek、Qwen、GLM、Kimi、Hermes、Llama、Mistral、Gemma 等多种 parser。SGLang 的设计更接近“模型格式 detector + 结构约束生成”。它在 `function_call` 模块中维护各模型 detector，并通过 `reasoning_parser` 把 thinking 内容从普通 content 中分离出来；同时结合 structural tag / JSON schema 约束生成，支持在 strict 工具调用场景下减少格式错误。

本地 serving benchmark 表明，在 `Qwen/Qwen3.5-4B`、OpenAI Chat API、thinking disabled、identical prompt 场景下，SGLang 与 vLLM 都能稳定完成 120 个 case、共 3720 个请求，失败数为 0。SGLang 平均 req/s 为 4.30，vLLM 为 3.17；SGLang 平均 TPOT 为 27.61 ms，vLLM 为 39.41 ms。这说明两者 serving 路径稳定，但该轮 benchmark 主要测的是 streaming、并发、prefix 复用与基础服务性能，并未直接测工具调用准确率。

在此基础上，新增的 vLLM 工具调用专项实验使用 `Qwen2.5-7B-Instruct + hermes tool parser`，显式构造 125 条工具调用 case 和 94 条 positive-only case，分别测工具选择、调用时机、参数 schema、参数语义、strict grammar 开销。结果显示：普通 auto tool-call 的 `tool_call_accuracy=0.752`、`schema_match_rate=0.9681`、`call_timing_accuracy=0.968`；开启 structured backend 本身几乎没有可测开销；在 auto tool-call 下加 `strict: true` 后平均延迟从 620.25 ms 到 626.86 ms，约增加 1.1%，同时 `false_call_rate` 从 0.129 降到 0.0968。也就是说，工具调用专项评测把“功能是否支持”进一步拆成了“是否选对工具、何时调用、参数是否合法、参数语义是否正确、约束解码是否带来开销”。

## 1. 协议层：工具调用到底是什么

### 1.1 OpenAI Function Calling

OpenAI Function Calling 的核心流程是：

```text
用户请求
  -> 模型根据 tools 判断是否需要工具
  -> 模型返回 tool_calls
  -> 应用侧执行真实函数
  -> 应用把 tool output 回传给模型
  -> 模型生成最终回答
```

其中 `tools` 是函数说明书，不是函数本身。一个 tool schema 通常包含：

| 字段 | 含义 |
| --- | --- |
| `type` | 工具类型，函数调用通常是 `function` |
| `name` | 函数名 |
| `description` | 函数用途，帮助模型判断何时调用 |
| `parameters` | JSON Schema，描述参数结构 |
| `strict` | 是否严格遵守 schema |

示例：

```json
{
  "type": "function",
  "name": "get_weather",
  "description": "Get current weather for a given city.",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "City name"
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"]
      }
    },
    "required": ["city", "unit"],
    "additionalProperties": false
  },
  "strict": true
}
```

模型返回的 `tool_calls` 是调用请求，不是执行结果：

```json
{
  "tool_calls": [
    {
      "id": "call_123",
      "type": "function",
      "function": {
        "name": "get_weather",
        "arguments": "{\"city\":\"Beijing\",\"unit\":\"celsius\"}"
      }
    }
  ]
}
```

这里最容易踩坑的是：`arguments` 通常是 JSON 字符串，应用侧需要解析、校验、执行，再把结果按 `tool_call_id` 回传。并行工具调用时，模型可能一次返回多个 `tool_calls`，应用侧不能只处理第一个。

### 1.2 Anthropic Tool Use

Anthropic 的 Tool Use 与 OpenAI Function Calling 在思想上相同，但消息结构不同。Claude 响应中可能出现 `tool_use` content block，应用执行工具后再发回 `tool_result` block。

可以把二者对应关系理解为：

| 概念 | OpenAI | Anthropic |
| --- | --- | --- |
| 工具定义 | `tools[].function` / function schema | `tools[]` |
| 模型请求调用 | `tool_calls` | `tool_use` block |
| 调用 ID | `tool_call.id` | `tool_use.id` |
| 工具结果 | `role=tool` 或 `function_call_output` | `tool_result` block |
| 多工具调用 | `tool_calls` 数组 | 多个 `tool_use` block |

因此，推理引擎若要同时兼容 OpenAI 和 Anthropic API，本质上需要做一层协议转换：把模型原生输出解析成内部统一结构，再渲染成目标 API 的响应格式。

## 2. 模型层：chat template 决定解析难度

工具调用解析不是一个纯 API 问题。开源模型通常不是“天然生成 OpenAI `tool_calls` JSON”，而是按各自 chat template 生成特殊标记、XML、JSON block 或 DSML block。推理引擎要做的是把这些模型原生文本转换成 OpenAI-compatible 字段。

### 2.1 Qwen 系列

Qwen 的关键点是 thinking 与 tool call 常常共存。Qwen3 / Qwen3.5 类模型通常使用 `<think>...</think>` 表示思考过程，并可通过 `chat_template_kwargs.enable_thinking=false` 关闭 thinking。工具调用格式则随模型版本变化，例如 Qwen2.5 与 Qwen3 Coder 的工具调用标记和参数组织方式并不完全相同。

这意味着推理引擎必须同时处理：

- 是否启用 thinking；
- thinking 内容是否应返回到 `reasoning_content`；
- tool call 是否出现在 `<tool_call>` 或其他模型专用结构中；
- 如果 thinking 与 tool call 连在一起，哪个 parser 先处理。

本地 benchmark 的 Qwen3.5 配置中明确设置：

```json
{
  "chat_template_kwargs": {
    "enable_thinking": false
  }
}
```

这保证了本轮性能测试主要测 content streaming，而不是 reasoning parser 开销。

### 2.2 DeepSeek 系列

DeepSeek 系列的复杂性在于不同版本的工具调用块与 reasoning 格式不同。本地 vLLM 源码中可以看到 `deepseekv3`、`deepseekv31`、`deepseekv32`、`deepseekv4` 等 parser；SGLang 也有对应 detector。以 DeepSeek V4 / V3.2 一类 DSML 风格为例，工具调用可能被包在模型专用标签中，再由 tokenizer / encoding 或 detector 转换为 OpenAI format。

因此，对 DeepSeek 这类模型，正确选择 parser 是必要条件。只传入 `tools` 不等于工具调用一定可用；如果 `--tool-call-parser` 与模型实际 template 不匹配，结果通常是 raw markup 残留在 `content` 中，或 `tool_calls` 为空。

### 2.3 GLM / Kimi / Mistral 等模型

GLM、Kimi、Mistral、Hermes 等模型也各有工具调用格式。SGLang 本地源码中的 `FunctionCallParser.ToolCallParserEnum` 已包含 `glm45`、`kimi_k2`、`mistral`、`hermes`、`gemma4`、`qwen3_coder` 等 detector；vLLM 的 `vllm/tool_parsers` 目录也包含对应 parser。

这说明当前推理引擎的工具调用支持正在从“少数模型硬编码”走向“模型格式库”。但这也带来一个工程现实：工具调用不是只看模型能力，还要看该模型的 template 是否被当前引擎版本正确适配。

## 3. 约束解码：为什么 XGrammar 重要

工具调用最终要落到结构化生成。普通解码只问“下一个 token 概率最高是什么”，而 grammar-constrained decoding 会在每一步加一层合法性检查：

```text
当前已生成前缀 + schema / grammar
  -> 计算下一步哪些 token 合法
  -> mask 掉非法 token
  -> 模型只能从合法 token 中采样
```

例如 schema 要求：

```json
{
  "city": "string",
  "unit": "celsius" | "fahrenheit"
}
```

当模型生成到 `"unit": "` 时，约束解码会屏蔽除 `celsius` / `fahrenheit` 之外的值。它不是生成完再检查，而是在生成过程中就不允许走到非法结构。

XGrammar 的定位是 efficient, flexible and portable structured generation。它支持 JSON、regex、自定义上下文无关文法，并已被 vLLM、SGLang、TensorRT-LLM、MLC-LLM 等集成。对工具调用而言，XGrammar / structural tag / JSON schema 这类约束的价值在于：

- 降低参数 JSON 语法错误；
- 降低多余字段、字段类型错误；
- 在 `tool_choice=required` 或 strict schema 下强制生成工具调用结构；
- 让 parser 不必完全依赖事后正则提取。

但它也不能解决所有问题。约束解码保证“结构合法”，不保证“工具选得对”或“参数语义正确”。例如它可以保证 `unit` 必须是枚举值，但不能保证用户问的是北京时模型不会填成上海。

## 4. vLLM 支持方式

### 4.1 启用方式

vLLM 的 OpenAI serving 层通过参数启用工具调用与 reasoning 解析：

```text
--enable-auto-tool-choice
--tool-call-parser <parser_name>
--reasoning-parser <parser_name>
```

本地源码 `week1/vllm/vllm/entrypoints/openai/cli_args.py` 中可以看到 `enable_auto_tool_choice`、`tool_call_parser` 等参数；`ParserManager.get_tool_parser` 会在启用 auto tool choice 且传入 parser 名称时加载对应 ToolParser。

### 4.2 Tool parser 架构

vLLM 的 `ToolParser` 抽象定义在：

```text
week1/vllm/vllm/tool_parsers/abstract_tool_parser.py
```

它包含两个核心接口：

| 接口 | 用途 |
| --- | --- |
| `extract_tool_calls` | 非 streaming 场景，一次性从完整输出中提取工具调用 |
| `extract_tool_calls_streaming` | streaming 场景，从增量 token / 文本中解析 tool call delta |

同一文件中还可以看到 `adjust_request`：当工具调用需要结构约束时，vLLM 会基于工具 schema 设置 `structured_outputs` 或 `json_schema`，从而把 tool calling 与 structured output 连接起来。

本地 `week1/vllm/vllm/tool_parsers` 目录中包含大量模型 parser，例如：

```text
deepseekv3_tool_parser.py
deepseekv32_tool_parser.py
deepseekv4_tool_parser.py
qwen3coder_tool_parser.py
qwen3xml_tool_parser.py
glm4_moe_tool_parser.py
kimi_k2_tool_parser.py
hermes_tool_parser.py
mistral_tool_parser.py
gemma4_tool_parser.py
openai_tool_parser.py
```

这说明 vLLM 的能力边界是“parser 覆盖到的模型格式”。如果模型 template 已适配，OpenAI-compatible `tool_calls` 可以由引擎统一返回；如果没有适配，则需要新增 parser 或使用更通用的 JSON / OpenAI parser。

### 4.3 Reasoning parser 架构

vLLM 的 reasoning 输出文档提示，`reasoning_content` 已迁移为 `reasoning`，但本地与下游兼容中仍能看到 `reasoning_content` 字段。源码中 `ParserManager` 支持同时加载 tool parser 与 reasoning parser，并将二者组合为统一 Parser。

这点很重要：真实模型输出可能是：

```text
<think>先分析问题...</think>
<tool_call>{...}</tool_call>
```

如果没有统一 parser pipeline，工具调用可能被误归入 reasoning，或 reasoning 标签残留在 content 里。vLLM 的 parser manager 正是在解决这种组合问题。

### 4.4 vLLM 的工程判断

vLLM 的优势是 OpenAI-compatible serving 层成熟，工具调用 parser 数量多，适合快速接入不同模型并暴露标准 API。风险是每个模型格式都需要 parser 适配，尤其 streaming 场景要处理半截 JSON、半截标签、工具调用 index、finish_reason 等细节。

对业务落地来说，vLLM 更适合作为标准化 API 网关式 serving engine：只要模型 parser 可用，就能比较自然地接入现有 OpenAI function calling 客户端。

## 5. SGLang 支持方式

### 5.1 启用方式

SGLang 的 OpenAI serving 层通过：

```text
--tool-call-parser <parser_name>
--reasoning-parser <parser_name>
```

启用工具调用和思考内容解析。本地 benchmark 中 SGLang 启动参数曾使用：

```text
--reasoning-parser qwen3
--skip-server-warmup
--disable-cuda-graph
```

在本轮正式矩阵中，客户端通过 `chat_template_kwargs.enable_thinking=false` 关闭 Qwen thinking，因此没有把 reasoning 内容纳入性能测量。

### 5.2 FunctionCallParser 与 detector

SGLang 的核心文件是：

```text
week1/sglang/python/sglang/srt/function_call/function_call_parser.py
```

其中 `FunctionCallParser` 维护了 parser 名称到 detector 的映射：

```text
qwen / qwen25 / qwen3_coder
deepseekv3 / deepseekv31 / deepseekv32 / deepseekv4
glm / glm45 / glm47
kimi_k2
mistral
hermes
gemma4
gpt-oss
...
```

它同样区分 streaming 与 non-streaming：

| 方法 | 用途 |
| --- | --- |
| `parse_non_stream` | 从完整文本中提取工具调用 |
| `parse_stream_chunk` | 增量解析 streaming chunk |
| `get_structure_constraint` | 根据 `tool_choice`、strict、parallel_tool_calls 生成结构约束 |

这里比普通 parser 多出一个关键能力：SGLang 不只是事后 parse，还会根据工具 schema 和模型结构 tag 生成约束，用于指导模型输出工具调用格式。

### 5.3 ReasoningParser

SGLang 的 reasoning parser 在：

```text
week1/sglang/python/sglang/srt/parser/reasoning_parser.py
```

其基础逻辑是用开始 / 结束 token 把思考内容与普通内容拆开。例如：

| 模型类型 | thinking 标记 |
| --- | --- |
| DeepSeek R1 | `<think>...</think>` 或无开始标记、以 `</think>` 结束 |
| Qwen3 | `<think>...</think>` |
| Kimi | `◁think▷...◁/think▷` |
| Kimi K2 | `<think>...</think>`，并可能被 tool call section 打断 |
| GLM-4.5 | `<think>...</think>`，遇到 `<tool_call>` 可切出 |
| GPT-OSS | Harmony / analysis channel 风格 |

SGLang 的 detector 里还显式处理了 thinking 与 tool call 的交界，例如 Kimi K2 可以在 closing `</think>` 前进入 tool call section，GLM-4.5 可以用 `<tool_call>` 作为从 reasoning 切到 normal/tool 的起点。这说明 reasoning parser 不是简单字符串 split，而是要考虑工具调用对 thinking 流程的打断。

### 5.4 SGLang 的工程判断

SGLang 的优势是把模型原生格式、结构约束和 runtime 执行结合得更紧。它特别适合需要结构化生成、工具调用、reasoning 分离和 prefix/cache 优化共同工作的场景。

风险是配置敏感：`--tool-call-parser`、`--reasoning-parser`、chat template、`enable_thinking`、模型版本必须匹配。一旦 parser 与 template 不一致，工具调用可能无法被正确抽取。例如一些模型文档会明确提示不要用旧 Qwen parser 去解析新 XML 风格 tool call。

## 6. vLLM 与 SGLang 特性矩阵

| 维度 | vLLM | SGLang | 结论 |
| --- | --- | --- | --- |
| OpenAI Chat API | 成熟支持 | 成熟支持 | 两者都可作为 OpenAI-compatible 服务 |
| Tool schema 输入 | 支持 `tools` / `tool_choice` | 支持 `tools` / `tool_choice` | 协议层基本一致 |
| 自动工具调用 | `--enable-auto-tool-choice` + parser | `--tool-call-parser` + detector | 都依赖模型 parser |
| 并行工具调用 | 协议字段支持，parser 需处理多 call | `parallel_tool_calls` 可影响 schema constraint | SGLang 在 constraint 生成处更显式 |
| Streaming tool call | `extract_tool_calls_streaming` | `parse_stream_chunk` | 两者都专门处理流式增量 |
| Reasoning 分离 | `--reasoning-parser`，字段向 `reasoning` 迁移 | `--reasoning-parser`，输出 `reasoning_content` | 字段命名需做客户端兼容 |
| 模型格式覆盖 | 多 parser 文件，覆盖 DeepSeek/Qwen/GLM/Kimi 等 | detector 枚举覆盖 DeepSeek/Qwen/GLM/Kimi 等 | 覆盖面都在快速扩展 |
| 结构约束 | structured outputs / JSON schema / structural tag | structural tag / JSON schema constraint | SGLang 与 constrained generation 联系更紧 |
| 配置风险 | parser 名称与模型不匹配 | parser、reasoning、template 三者不匹配 | 都需要模型级 smoke test |
| 性能实测 | 本轮平均 req/s 3.17，TPOT 39.41 ms | 本轮平均 req/s 4.30，TPOT 27.61 ms | 本轮 workload 下 SGLang 更优 |

| 8192 | 1.16 | 0.98 | 1.18x | 160.4 ms | 550.5 ms | 25.73 ms | 47.79 ms |

工具调用相关解读：

工具调用 / reasoning parser 通常发生在 decode 输出侧或 streaming 后处理侧；而本轮结果显示，在长 prompt 场景中，prefill 与 prefix cache 对 TTFT 的影响非常大。对于 agent / tool-use workload，系统往往包含长 system prompt、工具说明、历史工具调用结果和多轮上下文，因此 prompt prefix 复用能力会直接影响工具调用链路的首 token 延迟。

换句话说，工具调用能力不能只看 parser 是否支持，还要看引擎能否承受“长工具说明 + 多轮历史 + streaming 输出”的 serving 压力。

### 7.4 Structured smoke

`benchmark_results/structured_smoke/qwen35_benchmark_20260603_105048.structured.json` 是一个最小 smoke：

| 引擎 | batch | prompt | max_tokens | req/s | TTFT | TPOT | E2E | 失败 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| SGLang | 1 | 32 | 8 | 2.23 | 272.5 ms | 24.70 ms | 445.4 ms | 0 |
| vLLM | 1 | 32 | 8 | 7.11 | 80.1 ms | 8.12 ms | 137.0 ms | 0 |

这组数据规模很小，只能说明两个服务 endpoint 与 streaming 采集链路可用，不能代表正式性能结论。它的价值在于作为后续工具调用 benchmark 的最小模板：可以在同样框架中加入 tool schema、预期 tool name、预期 arguments、strict mode 和 parser 输出校验。

### 7.5 工具调用与 grammar-guided generation 开销实验

在 serving benchmark 之后，新增了一个 vLLM 工具调用专项实验，目标不是比较 SGLang 与 vLLM 的通用吞吐，而是回答更细的问题：在真实 OpenAI-compatible tool-call 请求下，vLLM 能否稳定产出 `tool_calls`，参数是否符合 schema，strict grammar 会不会带来准确率收益或延迟开销。

实验配置如下：

| 维度 | 配置 |
| --- | --- |
| 任务配置 | `benchmark-vllm-v.s-sglang/configs/tasks/week4-task5b-grammar-overhead-qwen25-7b-v2.json` |
| 临时并行配置 | `/nfs100/xiaoyixiao/tmp/week4-task5b-grammar-overhead-gpu{2,3,7}.json` |
| 模型 | `Qwen2.5-7B-Instruct` |
| 推理引擎 | vLLM |
| tool parser | `hermes` |
| GPU | A100-SXM4-40GB，GPU 2 / 3 / 7 并行 |
| 端口 | 8102 / 8103 / 8107 |
| 数据集 | `data/tool_call_eval/tool_call_cases_v2.jsonl`，125 条；`data/tool_call_eval/tool_call_cases_v2_positive.jsonl`，94 条 |
| 输出 | `results/summary/grammar_overhead_qwen25_7b_gpu*_20260702_2148.jsonl` |

这轮实验把“grammar-guided generation”拆成 5 个 case，逐层隔离变量：

| case | 含义 | 主要回答的问题 |
| --- | --- | --- |
| `full-auto-no-structured-no-strict` | auto tool-call，不开 structured backend，不加 strict | 原始 auto tool-call baseline |
| `full-auto-structured-no-strict` | auto tool-call，开启 structured backend，不加 strict | 只开 structured backend 是否有额外开销 |
| `full-auto-structured-strict` | auto tool-call，开启 structured backend，并给 tool schema 加 `strict: true` | 真实 auto tool-call 下 strict grammar 的收益和开销 |
| `positive-oracle-structured-no-strict` | 只测正例，并用 oracle 指定应调用的工具，不加 strict | 排除工具选择后，参数生成 baseline |
| `positive-oracle-structured-strict` | 只测正例，并用 oracle 指定应调用的工具，加 `strict: true` | 排除工具选择后，strict grammar 对参数生成的影响 |

核心结果如下：

| case | cases | accuracy | schema | timing | false call | avg latency | p50 | p95 | p99 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| auto / no structured / no strict | 125 | 0.7520 | 0.9681 | 0.9680 | 0.1290 | 620.2 ms | 510.7 ms | 1193.3 ms | 3003.6 ms |
| auto / structured / no strict | 125 | 0.7520 | 0.9681 | 0.9680 | 0.1290 | 620.3 ms | 510.4 ms | 1193.5 ms | 3006.6 ms |
| auto / structured / strict | 125 | 0.7600 | 0.9681 | 0.9760 | 0.0968 | 626.9 ms | 510.2 ms | 1055.8 ms | 3597.9 ms |
| positive oracle / structured / no strict | 94 | 0.7021 | 1.0000 | 1.0000 | n/a | 425.3 ms | 369.0 ms | 780.1 ms | 832.2 ms |
| positive oracle / structured / strict | 94 | 0.7021 | 1.0000 | 1.0000 | n/a | 430.7 ms | 382.0 ms | 779.7 ms | 829.1 ms |

从这组数据可以得到三个直接结论。

第一，只开启 vLLM structured outputs backend 本身没有带来可观测开销。`full-auto-no-structured-no-strict` 与 `full-auto-structured-no-strict` 的平均延迟几乎相同，约 620 ms；准确率、schema 匹配率、调用时机也完全一致。这说明至少在本轮 `Qwen2.5-7B-Instruct + hermes parser` 的串行请求下，server 侧打开 structured backend 不会自动降低性能，真正影响生成行为的是请求里的 strict schema 约束。

第二，在更接近真实产品的 auto tool-call 场景中，`strict: true` 的平均延迟从 620.3 ms 增加到 626.9 ms，约 +1.1%。这个开销很小，但不是零。与此同时，`tool_call_accuracy` 从 0.7520 到 0.7600，`call_timing_accuracy` 从 0.9680 到 0.9760，`false_call_rate` 从 0.1290 降到 0.0968。也就是说，本轮 strict grammar 的主要收益不是让 schema 从不合法变合法，而是略微改善了整体调用行为；但 `schema_match_rate` 本身没有提升，因为 no-strict baseline 已经达到 0.9681。

第三，在 positive oracle 参数生成场景中，加 strict 后平均延迟从 425.3 ms 到 430.7 ms，约 +1.3%；`tool_call_accuracy`、`schema_match_rate`、`argument_semantic_match` 基本不变。这说明当工具名已经被 oracle 固定、且 schema 本身已经稳定时，strict grammar 对参数语义正确性帮助有限。它能约束 JSON 结构，但不能保证城市、日期、金额等业务参数一定抽取正确。

### 7.6 新旧测试的变化与科学性

旧测试主要回答“服务能不能稳定跑、吞吐和延迟如何”。它的优点是 workload 更接近通用 serving 性能测试，覆盖 batch size、prompt length、max_tokens、streaming、prefix 复用等维度，适合比较 SGLang 与 vLLM 的基础性能。缺点是它没有传入真实 `tools`，也没有验证 `tool_calls`、schema、参数语义和调用时机，因此不能回答“工具调用是否正确”。

新增工具调用测试回答的是另一类问题：在明确的工具 schema 和期望答案下，模型是否应该调用工具、是否调用了正确工具、参数 JSON 是否符合 schema、参数值是否符合用户意图、strict grammar 是否带来准确率或延迟变化。它比旧测试更科学的地方在于：

- 把准确率拆成多个可解释指标，而不是只看请求是否成功。
- 把 auto 工具选择和 oracle 参数生成拆开，避免把“工具选错”和“参数生成错”混在一起。
- 把 structured backend 与 `strict: true` 拆开，避免把“server 开关开销”和“grammar 约束开销”混为一谈。
- 同时记录平均延迟、p50、p95、p99、token 均值和 token throughput，使“性能开销”有明确计算口径。
- 正例和正负混合集分开测，既能看真实端到端 auto tool-call，也能看参数生成的纯影响。

这并不意味着新测试完全替代旧测试。两者各自回答的问题不同：旧测试更适合做引擎 serving 性能横向比较，新测试更适合做工具调用正确性与结构约束开销评估。新测试的局限也需要保留：当前只有 vLLM + Qwen2.5-7B 的单次结果，且 5 组为了利用空闲 GPU 是跨 GPU 并行跑的，仍可能受到单次运行、GPU 状态、服务启动和缓存状态影响。正式结论建议增加 3-5 次 repeat，并在同一组 GPU 或同一生命周期下复跑，进一步降低偶然性。

## 8. 大模型启动结果对本主题的影响

`2026-06-20-large-model-startup-action-notes.md` 的核心结论是：当前阻塞点不是权重或评测框架，而是大模型在当前服务器、GPU 和启动参数下无法稳定启动。

### 8.1 Qwen2.5-14B

原始配置：

```text
max_model_len = 32768
TP = 1
gpu_memory_utilization = 0.80
GPU = 7
```

失败原因是 KV cache 预算不足。启动 smoke 已验证：

```text
max_model_len=16384, gpu_memory_utilization=0.95, TP=1 -> READY
max_model_len=8192, gpu_memory_utilization=0.90, TP=1 -> READY
```

对工具调用 benchmark 的影响：14B 可作为后续工具调用准确率测试模型，但建议先用 8192 或 16384 上下文，不要继续沿用 32768。

### 8.2 QwQ-32B

原始配置：

```text
max_model_len = 32768
TP = 2
gpu_memory_utilization = 0.80
GPU = 1,2
```

失败原因是权重与 CUDA graph 开销后 KV cache block 无可用内存。已验证可行：

```text
max_model_len=8192, gpu_memory_utilization=0.90, TP=2 -> READY
max_model_len=8192, gpu_memory_utilization=0.90, TP=4 -> READY
```

对工具调用 benchmark 的影响：32B 更适合作为 reasoning / thinking 解析测试对象，但正式矩阵应优先用 8192 上下文。若要更稳定并发，TP=4 比 TP=2 有更大 KV cache 余量。

### 8.3 GLM-4.5-Air-FP8

当前测试：

```text
max_model_len = 8192
TP = 4
gpu_memory_utilization = 0.90
tool_parser = glm45
GPU = 1,2,3,4
```

权重完整，47/47 shards 可加载，但在 FP8 Marlin / compressed-tensors / worker 初始化阶段失败。结论是：在 clean `/v1/models` startup smoke 通过前，不建议纳入正式工具调用对比结论。

这点对本报告很关键：GLM 系列在 parser 层有工具调用支持，不代表当前硬件和 runtime 组合已经具备可评测性。功能支持与可运行性必须分开判断。

## 9. 后续 Benchmark 设计建议

当前 benchmark 已经从基础 serving 性能扩展到 vLLM 工具调用专项评测，但覆盖面仍不完整：一方面，SGLang 还没有纳入同一套工具调用准确率对比；另一方面，grammar-guided generation 只在 Qwen2.5-7B 上做了单次实验。下一组 Benchmark 建议继续按以下维度扩展。

### 9.1 指标

| 指标 | 含义 |
| --- | --- |
| tool selection accuracy | 是否选择了正确工具 |
| argument schema valid rate | arguments 是否可解析且符合 JSON Schema |
| argument semantic accuracy | 参数值是否符合用户意图 |
| no-tool precision | 不该调用工具时是否避免调用 |
| parallel tool call accuracy | 多工具调用数量、顺序、参数是否正确 |
| reasoning separation accuracy | thinking 是否进入 `reasoning_content`，最终答案是否干净 |
| raw markup leakage rate | `<think>`、`<tool_call>`、DSML 等标记是否泄漏到 `content` |
| streaming assembly success | 流式 tool call delta 是否可正确拼装 |
| grammar overhead | strict schema / grammar-guided generation 对平均延迟、尾延迟和吞吐的影响 |

### 9.2 测试用例类型

| 类型 | 示例 | 检查点 |
| --- | --- | --- |
| 单工具调用 | “查询北京天气” | 工具名、城市参数、单位枚举 |
| 多工具并行 | “查北京和上海天气” | 返回两个 tool calls，参数分别正确 |
| 不应调用工具 | “解释什么是天气预报” | 不返回 tool_calls |
| schema 约束 | 单位只能 `celsius/fahrenheit` | enum 合法 |
| 缺参澄清 | “帮我订票”但未给日期 | 是否追问而不是瞎填 |
| reasoning + tool | 先思考再调用工具 | reasoning 与 tool_calls 不混淆 |
| 多轮工具结果 | 工具结果回传后继续回答 | tool_call_id / tool result 对齐 |

### 9.3 推荐模型与配置

| 模型 | 推荐用途 | 当前建议配置 |
| --- | --- | --- |
| Qwen/Qwen3.5-4B | 快速验证 parser、streaming、thinking toggle | 沿用当前 SGLang / vLLM endpoint |
| Qwen2.5-14B-Instruct | 中等规模工具调用准确率 | `max_model_len=8192/16384` |
| QwQ-32B | reasoning 分离与复杂任务 | `max_model_len=8192`，TP=2 或 TP=4 |
| GLM-4.5-Air-FP8 | 暂不纳入正式结论 | 先解决 FP8 runtime 启动 |

### 9.4 下一步建议

后续实验建议优先做三件事。

第一，补齐同一数据集上的 SGLang 对照。当前 `week4-task5b` 已经把工具调用准确率、schema、调用时机和 grammar 开销拆清楚，下一步应在 SGLang 上复用相同 case 和相同指标，避免继续用“vLLM 工具调用准确率”与“SGLang serving 吞吐”做间接比较。

第二，对 grammar overhead 做 repeat。当前 strict 开销约 1% 级别，但这是单次、跨 GPU 并行结果。正式报告应至少 repeat 3-5 次，输出均值、标准差和置信区间，并尽量固定 GPU、端口、server lifecycle，减少启动和缓存状态影响。

第三，增加 schema 难度分层。当前数据集中 schema 已经比较稳定，导致 strict 前后的 `schema_match_rate` 差异不明显。可以单独构造 simple / enum-heavy / nested object / nested array / required-missing 等 schema stress 组，观察 grammar 在复杂 schema 下是否真正降低 malformed JSON、多余字段、类型错误和 enum 错误。

## 10. 结论

第一，工具调用是协议、模型 template、推理引擎 parser 和应用执行层共同完成的系统能力。OpenAI / Anthropic 只定义了上层交互形状，开源模型真正落地时还依赖推理引擎把模型原生输出转换成标准 `tool_calls` 或 `tool_use`。

第二，vLLM 与 SGLang 都已经具备较完整的工具调用与 reasoning 解析框架。vLLM 更偏 OpenAI-compatible serving 与 parser registry；SGLang 更偏 detector、structural tag、grammar-guided generation 与 runtime 结合。两者的共同风险是 parser 与模型 chat template 不匹配。

第三，reasoning content 解析是工具调用之外的独立复杂度。`<think>...</think>`、analysis channel、DSML、XML tool block 等结构可能互相嵌套或打断，因此 parser 需要按模型格式维护状态机，而不能只做简单字符串切分。

第四，本地 serving benchmark 证明当前 Qwen3.5-4B 的 SGLang / vLLM OpenAI-compatible serving 路径稳定，且 SGLang 在 identical prompt 场景下吞吐和 TPOT 更优。但这组数据不能替代工具调用专项评测。新增的 vLLM + Qwen2.5-7B 工具调用实验进一步说明：普通 auto tool-call 已经能稳定产出结构化 `tool_calls`，`schema_match_rate` 达到 0.9681；在 structured backend 上开启 `strict: true` 的平均延迟开销约 1% 级别，能略微改善调用时机和 false call，但不能显著提升参数语义正确性。

第五，大模型启动笔记说明功能支持和可评测性必须分开看。Qwen2.5-14B 与 QwQ-32B 已有可行启动参数，适合进入下一轮工具调用 / reasoning benchmark；GLM-4.5-Air-FP8 虽有 parser 配置线索，但当前 runtime 尚未稳定启动，不应进入正式结论。
