# 调研报告 4：模型评测工具链与长上下文支持

日期：2026-06-30

## 1. 背景与目标

本周目标是梳理模型能力评测、推理性能评测和长上下文支持之间的关系，形成后续 Week 7-10 团队评测体系的工具选型建议，并产出一次能力 + 性能联合评测报告模板。

需要明确三类问题：模型会不会、服务快不快、长上下文是否真的可用。能力评测框架、推理引擎 benchmark 和长上下文专项 benchmark 分别回答不同问题，不能互相替代。

## 2. 评测工具链总览

| 类别 | 推荐工具 | 回答的问题 |
| --- | --- | --- |
| 能力评测 | lm-eval-harness、OpenCompass、HELM、HumanEval 类工具 | 模型在标准任务、内部任务或代码任务上答得是否正确 |
| 性能评测 | vLLM benchmark、llm-perf、Locust、自建 benchmark 框架 | 在线 serving 的 TTFT、TPOT、ITL、吞吐、并发和业务流量表现 |
| 长上下文评测 | LongBench、RULER、InfiniteBench | 模型是否能利用长距离上下文 |
| 长上下文推理机制 | StreamingLLM、sliding window attention、ring attention、vLLM/SGLang/TensorRT-LLM 等引擎能力 | 长输入/持续生成时如何管理 KV cache、上下文窗口、序列并行和延迟 |

## 3. lm-eval-harness

### 3.1 工具定位

lm-eval-harness 是通用能力评测框架，适合快速复现公开 benchmark、用 YAML 接入内部 task，并通过 HF/vLLM/SGLang/API 后端评测不同模型。

### 3.2 任务注册机制

任务生命周期可以概括为：发现任务 -> 索引任务 -> 构造任务 -> 加载 YAML -> 执行评测。`TaskManager` 负责搜索内置和外部路径，索引层暴露 task/group/tag，factory 与 YAML loader 构造可执行 task。

### 3.3 上手实践结果

已按用户确认将 `lm_eval[hf]` 安装到 `.venvs/lm-eval`，版本为 `0.4.12`，并验证 `lm-eval --help`、`lm-eval ls tasks` 与 `lm-eval validate --tasks arc_easy,hellaswag` 可正常执行。

本轮真实能力评测选择 `/nfs100/xiaoyixiao/models/Qwen2.5-7B-Instruct`，原因是它是标准 `Qwen2ForCausalLM` instruct 模型，本地权重完整且在单张 A100 上运行风险低于 14B/32B。正式任务使用指导书 fallback 数据集 `arc_easy,hellaswag`，0-shot，全量运行，并启用模型自带 chat template。

结果：`arc_easy` 样本数 2,376，`acc=0.6831`、`acc_norm=0.5088`；`hellaswag` 样本数 10,042，`acc=0.5707`、`acc_norm=0.6538`。原始 JSON 和完整日志保存在 `eval_runs/results/qwen25_7b_arc_easy_hellaswag_20260630_162918/` 与 `eval_runs/raw/qwen25_7b_arc_easy_hellaswag_20260630_162918/`。

### 3.4 优缺点与接入成本

优点是任务生态丰富、复现方便、YAML task 适合版本控制；缺点是工程化可视化较弱，复杂业务指标需要 Python 辅助函数。新模型接入难度中等，新数据集接入难度中等。

## 4. OpenCompass

### 4.1 工具定位

OpenCompass 更偏团队级评测平台，通过 Python config 管理模型、数据集、推理、评测和汇总。当前 `.venvs/opencompass` 可用，版本 `0.5.2`，并已完成一次对照实跑。

### 4.2 配置体系

OpenCompass 的流程是 Configure -> Inference -> Evaluation -> Visualization。`models` 定义模型，`datasets` 定义数据集，`infer_cfg` 定义 prompt/retriever/inferencer，`eval_cfg` 定义 evaluator 和 metric。

### 4.3 与 lm-eval-harness 对比

lm-eval 上手更快，适合公开任务复现；OpenCompass 配置更工程化，适合长期维护团队评测矩阵。建议两者并用：OpenCompass 做主矩阵，lm-eval 做快速对照和公开 benchmark 补充。

本次 OpenCompass 对照实跑使用 `Qwen2.5-7B-Instruct`、`ARC_e_gen` 与 `hellaswag_gen_6faab5`，通过 `HuggingFacewithChatTemplate` 走 generation 模式。OpenCompass summary 输出：`ARC-e accuracy=0.71`，`hellaswag accuracy=1.19`。该结果不能与 lm-eval 的 PPL/multiple-choice 分数严格等价比较，因为本次 OpenCompass 完成路径是生成式评测，且 predictions 显示模型经常输出解释性短语而不是直接输出 `A/B/C/D`，导致答案抽取失败。

过程上还暴露了 OpenCompass 接入风险：`HuggingFacewithChatTemplate` 不支持 PPL-based evaluation；当前 Transformers/Qwen2Tokenizer 与 OpenCompass 0.5.2 存在 `batch_encode_plus` 兼容问题，本次通过运行时 `sitecustomize.py` patch 解决。

## 5. vLLM benchmark

### 5.1 性能评测方法

`vllm bench serve` 面向在线 OpenAI-compatible 服务，能测 TTFT、TPOT、ITL、E2E latency、Requests/s 和 tokens/s。本周生成了 server 脚本、serving benchmark 脚本和 offline latency/throughput 脚本。

### 5.2 指标物理含义

TTFT 反映 prefill、排队和首轮 decode；TPOT/ITL 反映 decode 效率与流式体感；Requests/s 与 tokens/s 反映吞吐；E2E latency 反映完整请求耗时。Input length 主要增加 prefill 和 KV cache 压力，Output length 主要增加 decode 压力。

### 5.3 实测结果

已完成一次 vLLM OpenAI-compatible serving 实跑。服务使用 `.venvs/vllm` 中的 vLLM `0.22.0`，模型为 `/nfs100/xiaoyixiao/models/Qwen2.5-7B-Instruct`，served model name 为 `qwen25-7b-instruct`，单卡 GPU 4，`max_model_len=4096`，`gpu_memory_utilization=0.90`，TP=1。

三组 `vllm bench serve` workload 结果如下：

| workload | input/output | req/s | output tok/s | mean TTFT ms | mean TPOT ms | mean E2E ms |
| --- | --- | ---: | ---: | ---: | ---: | ---: |
| short_in_short_out | 128/128 | 3.7558 | 480.7383 | 85.40 | 12.74 | 1703.95 |
| long_in_short_out | 2048/128 | 3.6193 | 463.2690 | 288.56 | 22.62 | 3161.64 |
| long_in_long_out | 2048/512 | 1.9363 | 991.4025 | 78.77 | 14.13 | 7296.80 |

长输入短输出显著拉高 TTFT，说明 prefill 压力增加；长输入长输出的 E2E latency 最高，主要由更长 decode 阶段驱动。原始 JSON、server log 和 bench log 保存在 `perf_runs/results/vllm_qwen25_7b_serve_bench_20260702_033210/` 与 `perf_runs/raw/vllm_qwen25_7b_serve_bench_20260702_033210/`。

## 6. 长上下文支持

### 6.1 StreamingLLM 核心思想

StreamingLLM 观察到 attention sink 现象：初始 token 即使语义不重要，也会吸收大量注意力。纯 sliding window 丢弃初始 token 后模型会不稳定；保留初始 sink token KV + 最近窗口 token KV 可以在有界 KV cache 下稳定持续生成。

### 6.2 attention sink

attention sink 是注意力分布中的稳定锚点。StreamingLLM 利用它解决持续流式生成中的 KV cache 增长和窗口崩溃问题，但不扩展模型长期记忆，也不能让模型理解已丢弃的中间历史。

### 6.3 长上下文 benchmark 与推理引擎的关系

LongBench/RULER/InfiniteBench 评估长上下文能力；vLLM 评估长输入 serving 性能；StreamingLLM 是流式推理机制。三者处在不同层次，应联合而不是混用。

## 7. 工具矩阵

完整矩阵见 `tool_matrix/eval_tool_matrix.md`。结论是：能力评测主框架建议 OpenCompass，公开任务快速复现保留 lm-eval，代码能力专项补充 HumanEval/EvalPlus；性能压测使用 vLLM benchmark，并补充 llm-perf、Locust 和自建 benchmark；长上下文专项使用 LongBench/RULER/InfiniteBench，同时补充 sliding window、ring attention 和不同推理引擎的长上下文支持差异。

## 8. 能力 + 性能联合评测结果

当前联合评测报告中，lm-eval 能力评测已完成真实全量 run，OpenCompass 已完成生成式对照实跑，vLLM serving 性能评测也已完成三组 workload。能力和性能部分均保留模型、硬件、软件、启动参数、评测参数、原始 JSON 和完整日志路径。

## 9. Week 7-10 评测体系建议

1. 先固化 vLLM serving benchmark，建立性能基线和启动参数记录。
2. 将本轮 lm-eval run 固化为能力评测基线，并补充更多模型的同配置对照。
3. 用 OpenCompass 组织团队主评测矩阵。
4. 补充 HumanEval/EvalPlus/BigCodeBench 等代码能力评测专题。
5. 用 llm-perf、Locust 和自建 benchmark 框架补足 API/业务场景压测。
6. 引入 LongBench/RULER 做长上下文专项，并补充 sliding window/ring attention 机制分析。
7. 对 vLLM/SGLang/TensorRT-LLM/LMDeploy 做长上下文 serving 支持差异矩阵。
8. 每次报告必须同时记录模型、硬件、软件、启动参数、评测参数、原始输出路径和结论。

## 10. 风险与后续工作

- 能力评测已有 `Qwen2.5-7B-Instruct` 的 `arc_easy/hellaswag` 真实基线；OpenCompass 也已有生成式对照，但该分数受输出格式抽取影响，不能与 lm-eval 分数严格等价比较。
- HumanEval 类代码评测、llm-perf/Locust、自建 benchmark、ring attention、推理引擎长上下文差异仍需补充专题阅读和实测。
- vLLM serving 性能已有基线，但只覆盖 `Qwen2.5-7B-Instruct` 单模型、单卡、三组 synthetic workload；后续仍需补多模型、多并发、多输入长度和长时间稳定性测试。
- 历史 benchmark 结果来自不同任务/模型/参数，不能直接混入本报告表格。
- 长上下文能力与长输入性能必须分开解释。
